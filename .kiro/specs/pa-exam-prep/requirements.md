# Requirements Document

## Introduction

The PA Exam Prep App is an adaptive, AI-powered study platform for Physician Assistant students preparing for the PANCE (entry-level) and PANRE (recertification) exams. The platform identifies knowledge gaps through an Elo-style scoring model, serves rich multimedia content (images, video embeds, PDFs), and personalizes study sessions using the Claude AI API. The system is a React PWA hosted on Vercel with a Node.js/Express API and PostgreSQL database on Railway.

## Glossary

- **App**: The PA Exam Prep React Progressive Web Application
- **API**: The Node.js + Express backend service deployed on Railway
- **Student**: A user with the `student` role who studies for PANCE or PANRE exams
- **Admin**: A user with the `admin` role who manages the question bank, users, and media
- **Question_Bank**: The collection of all questions stored in the `questions` table
- **Question**: A single exam-style question with a stem/vignette, answer options, explanation, and optional media
- **Attempt**: A single answer submission by a Student for a Question within a Study_Session
- **Study_Session**: A timed block of questions presented to a Student in one of three modes
- **Adaptive_Session**: A Study_Session where the AI_Engine selects questions based on gap remediation and strength reinforcement
- **Exam_Simulation**: A Study_Session consisting of a fixed 120-question PANCE-style timed block
- **Weak_Spot_Sprint**: A Study_Session consisting of a 10-question burst focused on a single flagged category
- **AI_Engine**: The adaptive engine module in the API that calls the Claude API to plan study sessions
- **Elo_Score**: A numeric rating per Student per Category, starting at 1000, adjusted based on question difficulty and correctness
- **Topic_Score**: A record in the `topic_scores` table representing a Student's Elo_Score for a specific Category
- **Category**: An NCCPA blueprint category (e.g., Cardiology) stored in the `categories` table
- **Readiness_Score**: A 0–100 score calculated from weighted Elo_Scores across all Categories, adjusted for exam date proximity
- **Diagnostic_Baseline**: A 20-question initial assessment spanning all NCCPA categories, completed by new Students
- **Gap_Category**: A Category where the Student's error rate exceeds 40% over the last 10 Attempts or the Elo_Score has declined over the last 3 Study_Sessions
- **Heatmap**: A visual grid of NCCPA blueprint Categories color-coded and labeled by mastery level
- **Bookmark**: A saved reference to a Question for later review
- **Media_Asset**: An image, audio clip, video embed, or PDF associated with a Question
- **Signed_URL**: A time-limited URL (15-minute TTL) generated by the API for accessing Media_Assets in Vercel Blob
- **Refresh_Token**: A rotating token stored in an httpOnly cookie used to obtain new JWTs
- **JWT**: A short-lived (15-minute) JSON Web Token used for API authentication
- **NCCPA_Blueprint**: The official content blueprint published by the NCCPA defining exam topic areas and weightings
- **Decay_Factor**: A multiplier applied nightly to Topic_Scores for Categories not recently reviewed, implementing spaced repetition
- **Case**: A top-level grouping entity representing one LITFL page, containing clinical context, one or more Sub_Cases, Clinical_Pearls, Case_References, and Case_Tags
- **Sub_Case**: A subdivision within a Case representing a distinct ECG scenario; each Sub_Case has its own media and progressive Questions
- **ECG_Finding**: A structured finding within an answer, grouped by category (e.g., "Signs of inferior STEMI") with an array of individual finding strings
- **Clinical_Pearl**: A high-yield teaching point associated with a Case, stored as a string
- **Case_Reference**: An academic citation associated with a Case, with a citation string and optional URL
- **Case_Tag**: A classification tag set on a Case including primary_topic, secondary_topics, litfl_category, difficulty, board_relevance, and clinical_urgency
- **Source_Type**: The LITFL collection a Case originates from: `top_150_ecg`, `ecg_exigency`, `clinical_case`, or `ecg_library`
- **Question_Format**: The type of question within a Sub_Case: `describe_and_interpret`, `what_is_diagnosis`, `identify_features`, `clinical_decision`, `differential_diagnosis`, `algorithm_application`, or `compare_ecgs`
- **Media_Timing**: When a media item was captured relative to the clinical scenario: `initial`, `post_treatment`, `serial`, or `comparison`
- **LITFL_Import_Schema**: The Zod validation schema for the LITFL Case-based JSON import format, matching the ECG Quiz Question Bank schema v2

## Requirements

### Requirement 1: Google OAuth Authentication

**User Story:** As a Student, I want to sign in with my Google account, so that I can access the App without managing a separate password.

#### Acceptance Criteria

1. WHEN a Student taps "Sign in with Google", THE App SHALL redirect the browser to the Google OAuth consent screen.
2. WHEN Google returns a valid ID token, THE App SHALL send the ID token to the API at `POST /auth/google` for verification.
3. WHEN the API receives a valid Google ID token, THE API SHALL verify the token using `google-auth-library`, create or retrieve the user record, mint a JWT with a 15-minute expiry, and return a Refresh_Token in an httpOnly cookie.
4. WHEN a Refresh_Token is used to obtain a new JWT, THE API SHALL rotate the Refresh_Token and invalidate the previous one.
5. WHEN a Student signs in for the first time, THE API SHALL create a new user record with the `student` role and `free` plan.
6. IF the Google ID token verification fails, THEN THE API SHALL return a 401 Unauthorized response with a descriptive error message.
7. IF a blocked Student attempts to sign in, THEN THE API SHALL return a 403 Forbidden response with the message "Your account has been suspended."
8. THE API SHALL store all active sessions in the `sessions` table in PostgreSQL with a hashed Refresh_Token and expiration timestamp.

### Requirement 2: Role-Based Access Control

**User Story:** As an Admin, I want role-protected routes, so that only authorized users can access administrative functions.

#### Acceptance Criteria

1. THE API SHALL enforce JWT verification middleware on all authenticated routes.
2. THE API SHALL enforce role-check middleware on all `/admin/*` routes, permitting access only to users with the `admin` role.
3. IF a Student attempts to access an `/admin/*` route, THEN THE API SHALL return a 403 Forbidden response.
4. WHEN a new user is created, THE API SHALL assign the `student` role by default.
5. THE `users` table SHALL include a `plan` field with values `free` or `pro`, defaulting to `free`, to support future paid subscription tiers without schema migration.

### Requirement 3: Diagnostic Baseline Assessment

**User Story:** As a new Student, I want to complete an initial diagnostic assessment, so that the AI_Engine can establish my starting knowledge level across all exam categories.

#### Acceptance Criteria

1. WHEN a new Student completes sign-in for the first time, THE App SHALL present a 20-question Diagnostic_Baseline spanning all NCCPA Categories.
2. WHEN the Student completes the Diagnostic_Baseline, THE AI_Engine SHALL calculate an initial Elo_Score for each Category based on the Student's responses.
3. THE Diagnostic_Baseline SHALL include questions from each NCCPA Category to establish a Topic_Score for every Category.
4. WHILE the Student has not completed the Diagnostic_Baseline, THE App SHALL restrict access to Adaptive_Sessions and Exam_Simulations.

### Requirement 4: Elo-Style Scoring and Topic Scores

**User Story:** As a Student, I want my performance tracked per topic using an adaptive scoring model, so that the system can identify my strengths and weaknesses accurately.

#### Acceptance Criteria

1. THE API SHALL maintain one Topic_Score record per Student per Category in the `topic_scores` table.
2. WHEN a Student answers a Question correctly that has a higher difficulty than the Student's current Elo_Score predicts, THE API SHALL increase the Student's Elo_Score for that Category by a larger increment.
3. WHEN a Student answers a Question incorrectly that has a lower difficulty than the Student's current Elo_Score predicts, THE API SHALL decrease the Student's Elo_Score for that Category by a larger decrement.
4. THE API SHALL initialize new Topic_Scores with an Elo_Score of 1000.
5. THE Elo_Score recalculation cron job SHALL run nightly at 2:00 AM UTC and recalculate all Topic_Scores from Attempt history.

### Requirement 5: Gap Detection

**User Story:** As a Student, I want the system to identify my weak areas automatically, so that I can focus my study time on topics where I need the most improvement.

#### Acceptance Criteria

1. WHEN a Student's error rate exceeds 40% over the last 10 Attempts in a Category, THE API SHALL flag that Category as a Gap_Category.
2. WHEN a Student's Elo_Score for a Category has declined over the last 3 Study_Sessions, THE API SHALL flag that Category as a Gap_Category.
3. WHEN Gap_Categories are identified, THE App SHALL display the flagged Categories as "Focus Areas" on the Student's Dashboard.

### Requirement 6: Spaced Repetition Decay

**User Story:** As a Student, I want the system to prioritize topics I have not reviewed recently, so that I retain knowledge over time through spaced repetition.

#### Acceptance Criteria

1. THE spaced repetition decay cron job SHALL run nightly at 2:00 AM UTC and apply the Decay_Factor to Topic_Scores for Categories not recently reviewed.
2. WHEN a Category has not been reviewed for a longer period, THE decay cron job SHALL apply a larger Decay_Factor reduction to that Category's Topic_Score.
3. WHEN the AI_Engine plans a Study_Session, THE AI_Engine SHALL factor in time since last review for each Category to prioritize decayed topics.

### Requirement 7: AI-Powered Session Planning

**User Story:** As a Student, I want AI-planned study sessions tailored to my performance, so that each session targets my specific knowledge gaps and reinforces my strengths.

#### Acceptance Criteria

1. WHEN a Student starts an Adaptive_Session, THE AI_Engine SHALL call the Claude API with the Student's current Topic_Scores, Gap_Categories, and time-since-last-review data to generate a session plan.
2. THE AI_Engine SHALL return a session plan as a JSON object containing recommended Question IDs and difficulty targets.
3. WHEN a session plan is generated, THE API SHALL store the plan in the `study_sessions.ai_plan` JSONB field.
4. THE AI_Engine SHALL balance gap remediation and strength reinforcement in Adaptive_Session plans.

### Requirement 8: Study Session Modes

**User Story:** As a Student, I want to choose between different study modes, so that I can study in the way that fits my current needs.

#### Acceptance Criteria

1. THE App SHALL offer three Study_Session modes: Adaptive_Session, Exam_Simulation, and Weak_Spot_Sprint.
2. WHEN a Student selects Exam_Simulation mode, THE App SHALL present a fixed block of 120 questions in a timed format matching the PANCE exam structure.
3. WHEN a Student selects Weak_Spot_Sprint mode, THE App SHALL present a 10-question burst focused on a single Gap_Category selected by the Student.
4. WHEN a Student selects Adaptive_Session mode, THE AI_Engine SHALL generate a personalized session plan based on the Student's performance data.
5. WHEN a Study_Session starts, THE API SHALL create a `study_sessions` record with the session mode, start timestamp, and AI plan (if applicable).
6. WHEN a Study_Session ends, THE API SHALL record the end timestamp in the `study_sessions` record.

### Requirement 9: Question Interface and Answer Submission

**User Story:** As a Student, I want a clean, distraction-free question interface with immediate feedback, so that I can focus on learning from each question.

#### Acceptance Criteria

1. THE App SHALL display each Question in a single-question view with the question stem, answer options, and any associated Media_Assets.
2. WHEN a Question has associated Media_Assets, THE App SHALL display the media inline with a tap-to-expand interaction for images and EKGs.
3. WHEN a Student selects an answer option, THE App SHALL allow the Student to optionally provide a confidence self-rating on a 1–3 scale before submitting.
4. WHEN a Student submits an answer, THE API SHALL create an Attempt record with the selected option, correctness, duration in milliseconds, and optional confidence rating.
5. WHEN an answer is submitted, THE App SHALL immediately reveal the rationale, highlighting the correct answer with a green background and checkmark icon, and marking incorrect selections with an X icon.
6. WHEN an answer is submitted, THE App SHALL display the full explanation including distractor explanations for all answer options.
7. THE App SHALL provide a "Flag for Review" action on each Question during a Study_Session.
8. THE App SHALL provide a "Bookmark" action on each Question to save the Question for later review.

### Requirement 10: Readiness Score

**User Story:** As a Student, I want to see a single readiness score reflecting my overall exam preparedness, so that I can gauge how close I am to being ready for the exam.

#### Acceptance Criteria

1. THE API SHALL calculate a Readiness_Score from 0 to 100 based on the Student's weighted Elo_Scores across all NCCPA Categories.
2. WHEN a Student has set a target exam date, THE API SHALL adjust the Readiness_Score calculation to account for exam date proximity.
3. THE App SHALL display the Readiness_Score prominently on the Dashboard.

### Requirement 11: Dashboard

**User Story:** As a Student, I want a dashboard showing my study progress at a glance, so that I can quickly understand where I stand and what to study next.

#### Acceptance Criteria

1. THE App SHALL display the Student's Readiness_Score prominently on the Dashboard.
2. THE App SHALL display a "Today's Focus" section with 3 AI-recommended topic cards based on Gap_Categories and spaced repetition priorities.
3. THE App SHALL display a study streak counter that tracks consecutive days of study activity.
4. THE App SHALL implement grace days in the study streak counter to avoid resetting the streak when a Student misses a single day.
5. WHEN a Student has set a target exam date, THE App SHALL display an exam countdown showing days remaining.
6. THE App SHALL provide a link on the Dashboard to start an Adaptive_Session for the current day.

### Requirement 12: Progress Heatmap

**User Story:** As a Student, I want a visual heatmap of my performance across all exam categories, so that I can quickly identify strong and weak areas.

#### Acceptance Criteria

1. THE App SHALL display a Heatmap grid organized by NCCPA_Blueprint Categories, with each cell color-coded by mastery level.
2. THE App SHALL supplement color coding with text labels or patterns on each Heatmap cell so that mastery level is not conveyed by color alone.
3. WHEN a Student taps a Category cell on the Heatmap, THE App SHALL navigate to a detail view showing that Category's performance history, Elo_Score trend, and recent Attempts.
4. THE Heatmap SHALL use colors that meet a minimum contrast ratio of 3:1 against adjacent cells and background.

### Requirement 13: Analytics View

**User Story:** As a Student, I want detailed analytics on my study performance, so that I can track my improvement over time and understand my predicted exam outcome.

#### Acceptance Criteria

1. THE App SHALL display a performance-over-time chart showing the Student's accuracy trend across Study_Sessions.
2. THE App SHALL display a breakdown of the Student's weakest and strongest Categories based on Topic_Scores.
3. THE App SHALL display a predicted PANCE score band derived from the Student's current Readiness_Score and Topic_Scores.
4. THE App SHALL display summary statistics including total questions attempted, overall accuracy rate, and total study time.
5. THE analytics rollup cron job SHALL run nightly at 3:00 AM UTC and aggregate daily Attempt statistics per Student.

### Requirement 14: Content Library

**User Story:** As a Student, I want to browse the question bank, video library, and reference materials, so that I can study specific topics on demand.

#### Acceptance Criteria

1. THE App SHALL provide a browseable Question_Bank organized by organ system and Category.
2. THE App SHALL provide a video library section displaying YouTube and Vimeo embeds organized by topic.
3. THE App SHALL provide a "My Bookmarks" section listing all Questions the Student has bookmarked.
4. THE App SHALL provide a reference cards section with a PDF viewer for high-yield reference materials.
5. WHEN a Student searches the Question_Bank, THE API SHALL perform full-text search using PostgreSQL tsvector on the `questions` table.

### Requirement 15: Media Management and Delivery

**User Story:** As a Student, I want to view medical images, videos, and reference documents alongside questions, so that I can study with rich multimedia content.

#### Acceptance Criteria

1. WHEN the App requests a Media_Asset stored in Vercel Blob, THE API SHALL generate a Signed_URL with a 15-minute TTL.
2. THE API SHALL serve images through Vercel's image optimization pipeline, providing automatic WebP conversion, responsive srcset generation, and lazy loading support.
3. THE App SHALL render video content via YouTube or Vimeo iframe embeds without storing video files.
4. THE App SHALL display attribution text for all CC-licensed Media_Assets as recorded in the `question_media.attribution` field.
5. THE `question_media` table SHALL require a non-empty `alt_text` field for every Media_Asset record.

### Requirement 16: Admin Media Upload

**User Story:** As an Admin, I want to upload and manage media assets, so that I can attach images, audio, and PDFs to questions in the question bank.

#### Acceptance Criteria

1. WHEN an Admin initiates a media upload, THE API SHALL generate a presigned Vercel Blob upload URL.
2. WHEN the presigned URL is generated, THE App SHALL upload the file directly from the Admin's browser to Vercel Blob.
3. WHEN the upload completes, THE API SHALL create a `question_media` record with the Blob URL, media type, alt text, and attribution.
4. THE Admin interface SHALL require the Admin to provide alt text and attribution before completing a media upload.

### Requirement 17: Admin User Management

**User Story:** As an Admin, I want to manage user accounts, so that I can monitor usage and handle account issues.

#### Acceptance Criteria

1. THE Admin interface SHALL display a list of all user accounts showing name, email, join date, plan, and last active timestamp.
2. WHEN an Admin blocks a user, THE API SHALL set `is_blocked = true` on the user record, and the blocked Student SHALL receive a 403 response on subsequent sign-in attempts.
3. WHEN an Admin unblocks a user, THE API SHALL set `is_blocked = false` on the user record, restoring the Student's ability to sign in.
4. THE Admin interface SHALL display usage reports including active user count, total Study_Session count, and total Attempt volume.
5. THE Admin interface SHALL allow an Admin to view any Student's Topic_Scores, Readiness_Score, and Attempt history.

### Requirement 18: Admin Question Bank Management

**User Story:** As an Admin, I want to create, edit, and deactivate questions, so that I can maintain a high-quality question bank.

#### Acceptance Criteria

1. THE Admin interface SHALL provide a form to create new Questions with a stem, question type, difficulty (1–5), Category, explanation, and answer options.
2. THE Admin interface SHALL provide a form to edit existing Questions and their answer options.
3. WHEN an Admin deactivates a Question, THE API SHALL set `is_active = false` on the Question record.
4. WHILE a Question has `is_active = false`, THE API SHALL exclude that Question from all Student-facing queries and session plans.
5. THE API SHALL retain deactivated Questions in the database for historical Attempt records.

### Requirement 19: Request Validation

**User Story:** As a developer, I want all API inputs validated with a schema, so that the API rejects malformed requests before processing.

#### Acceptance Criteria

1. THE API SHALL validate all incoming request bodies and query parameters using Zod schemas.
2. IF a request fails Zod validation, THEN THE API SHALL return a 400 Bad Request response with a descriptive validation error message.

### Requirement 20: PWA and Offline Support

**User Story:** As a Student, I want to install the app on my phone and study offline, so that I can study without a WiFi connection.

#### Acceptance Criteria

1. THE App SHALL be installable on iOS and Android via the browser "Add to Home Screen" prompt.
2. THE App SHALL use a service worker registered via the Vite PWA plugin to cache application assets for offline access.
3. WHEN a Student starts a Study_Session, THE App SHALL cache the active question set in IndexedDB via Dexie.js so the Student can continue studying without a network connection.
4. WHEN the App is offline and the Student navigates to a non-cached route, THE App SHALL display a graceful offline fallback screen.
5. WHEN the App regains network connectivity after an offline Study_Session, THE App SHALL sync cached Attempt data to the API.

### Requirement 21: Mobile-First Responsive Layout

**User Story:** As a Student, I want the app to work well on my phone, tablet, and desktop, so that I can study on any device.

#### Acceptance Criteria

1. THE App SHALL use a mobile-first layout with Tailwind CSS breakpoints (`sm:`, `md:`, `lg:`) scaling up to tablet and desktop viewports.
2. THE App SHALL display a bottom tab bar with 4 tabs (Dashboard, Study, Library, Progress) that is thumb-reachable on all phone sizes.
3. THE App SHALL render in light mode only with no dark mode toggle.

### Requirement 22: Accessibility Compliance

**User Story:** As a Student with accessibility needs, I want the app to be usable with assistive technologies, so that I can study effectively regardless of ability.

#### Acceptance Criteria

1. THE App SHALL maintain a minimum color contrast ratio of 4.5:1 for all body text and 3:1 for large text and UI components.
2. THE App SHALL convey no information by color alone; all color indicators (correct/incorrect answers, Heatmap mastery levels) SHALL be accompanied by icons, labels, or patterns.
3. THE App SHALL make all interactive elements reachable and operable via keyboard navigation.
4. THE App SHALL display visible focus indicators on all interactive elements and SHALL NOT remove focus outlines without providing a custom visible replacement.
5. THE App SHALL provide descriptive `alt` text on all `<img>` elements and media, sourced from the `question_media.alt_text` field.
6. THE App SHALL associate all form inputs with `<label>` elements.
7. THE App SHALL announce form validation errors to screen readers via `aria-live` regions.
8. THE App SHALL ensure all interactive elements have a minimum touch target size of 44×44 pixels.
9. WHERE audio playback controls are displayed, THE App SHALL provide visible, labeled controls with ARIA labels.

### Requirement 23: Question Bank JSON Import/Export Template

**User Story:** As an Admin, I want a standardized JSON template for importing and exporting questions, so that I can bulk-manage the question bank without using the UI for every question.

#### Acceptance Criteria

1. THE API SHALL define a documented JSON schema for Question_Bank import/export that supports both the flat question format (question stem, type, difficulty, Category name, explanation, answer options, NCCPA task area, optional media) and the LITFL Case-based format (Cases with Sub_Cases, progressive Questions, structured ECG_Findings, Clinical_Pearls, Case_References, and Case_Tags).
2. THE API SHALL expose a `POST /admin/questions/import` endpoint that accepts an array of Questions in the flat JSON template format, validates each entry against the schema using Zod, and creates the corresponding Question, question_options, and question_media records.
3. IF any Question in an import batch fails validation, THEN THE API SHALL reject the entire batch and return a 400 response listing all validation errors with the index of each failing entry.
4. THE API SHALL expose a `GET /admin/questions/export` endpoint that returns all active Questions in the JSON template format, including their answer options and media references.
5. THE export endpoint SHALL support an optional `category_id` query parameter to filter the export to a single Category.
6. THE Admin interface SHALL provide an "Import Questions" action that accepts a JSON file upload and submits it to the import endpoint.
7. THE Admin interface SHALL provide an "Export Questions" action that downloads the current Question_Bank as a JSON file.
8. THE API SHALL expose a `POST /admin/cases/import` endpoint that accepts a LITFL Case-based JSON payload (with metadata and cases array), validates it against the LITFL_Import_Schema using Zod, and creates the corresponding Case, Sub_Case, Question, ECG_Finding, Clinical_Pearl, Case_Reference, Case_Tag, and Question_Media records.
9. IF any Case in a LITFL import batch fails validation, THEN THE API SHALL reject the entire batch and return a 400 response listing all validation errors with the case_id of each failing entry.
10. THE API SHALL expose a `GET /admin/cases/export` endpoint that returns all Cases in the LITFL Case-based JSON format, including Sub_Cases, Questions with structured answers, ECG_Findings, Clinical_Pearls, Case_References, Case_Tags, and Media with extended metadata.
11. THE export endpoint SHALL support optional `source_type`, `primary_topic`, and `difficulty` query parameters to filter the Case export.
12. FOR ALL valid Case-based JSON payloads, importing via `POST /admin/cases/import` then exporting via `GET /admin/cases/export` then importing again SHALL produce equivalent database records (round-trip property).

### Requirement 24: Alternative Answer Formats (Free Text and Audio)

**User Story:** As a Student, I want to choose between multiple choice, free text, or audio input when answering questions, so that I can practice recall and articulation beyond simple recognition.

#### Acceptance Criteria

1. THE App SHALL allow the Student to select an answer format before starting a Study_Session: multiple choice (default), free text, or audio input.
2. WHEN the Student selects free text format, THE App SHALL display the question stem without answer options and provide a text input field for the Student to type their answer.
3. WHEN the Student selects audio input format, THE App SHALL display the question stem without answer options and provide a record button for the Student to speak their answer.
4. WHEN the Student submits a free text answer, THE API SHALL send the Student's text response and the Question's correct answer and explanation to the Claude API for semantic evaluation.
5. WHEN the Student submits an audio answer, THE API SHALL first transcribe the audio to text using a speech-to-text service, then send the transcribed text and the Question's correct answer and explanation to the Claude API for semantic evaluation.
6. THE Claude API evaluation SHALL return a correctness judgment (correct, partially correct, or incorrect), a confidence score, and a brief explanation of what the Student got right and what was missed.
7. WHEN the AI evaluation returns "partially correct", THE API SHALL treat the Attempt as incorrect for Elo_Score calculations but SHALL display the partial credit feedback to the Student.
8. WHEN the AI evaluation completes, THE App SHALL display the AI's feedback alongside the full rationale, showing the Student's response compared to the expected answer.
9. THE Attempt record SHALL store the answer format used (`multiple_choice`, `free_text`, or `audio`) and, for free text and audio formats, the Student's raw response text.
10. THE App SHALL allow the Student to switch answer format between Study_Sessions but SHALL NOT allow switching mid-session.

### Requirement 25: Case-Based Question Organization

**User Story:** As a Student, I want questions organized into clinical cases with sub-cases, so that I can study ECG interpretation in realistic clinical scenarios with progressive complexity.

#### Acceptance Criteria

1. THE API SHALL store Cases as top-level entities with a unique case_id following the pattern `LITFL-ECG-XXXX`, `LITFL-EX-XXXX`, or `LITFL-CC-XXXX`.
2. THE API SHALL associate each Case with a source_url, Source_Type, title, clinical_context, and an array of authors.
3. THE API SHALL support one or more Sub_Cases per Case, each with a unique sub_case_id, optional sub_case_label, and optional sub_case_context.
4. WHEN a Case contains multiple Sub_Cases, THE App SHALL present each Sub_Case as a distinct scenario within the parent Case.
5. THE API SHALL associate each Sub_Case with its own set of Media and progressive Questions.
6. WHEN a Student browses Cases, THE App SHALL display the Case title, clinical_context, Source_Type, and difficulty tag.

### Requirement 26: Progressive Question Sequencing

**User Story:** As a Student, I want questions within a case presented in a defined sequence, so that I can build understanding progressively from description to diagnosis to clinical decision-making.

#### Acceptance Criteria

1. THE API SHALL store a sequence number on each Question within a Sub_Case, starting at 1.
2. THE API SHALL store a Question_Format enum on each Question to categorize the question type.
3. WHEN a Student studies a Sub_Case, THE App SHALL present Questions in ascending sequence order.
4. THE API SHALL associate each Question with zero or more related_media_ids referencing Media within the same Sub_Case.
5. WHEN a Question has related_media_ids, THE App SHALL display the referenced Media alongside the question stem.

### Requirement 27: Structured ECG Findings and Answer Format

**User Story:** As a Student, I want structured ECG findings in answers organized by category, so that I can learn systematic ECG interpretation with clear hierarchical findings.

#### Acceptance Criteria

1. THE API SHALL store a summary field on each Question answer containing a one-line diagnosis or key conclusion.
2. THE API SHALL store ECG_Findings as an array of category-grouped findings, where each entry has a category label and an array of finding strings.
3. THE API SHALL store an optional interpretation_text field on each Question answer for prose explanations.
4. THE API SHALL store an optional array of related_links on each Question answer, where each link has a text label and URL.
5. WHEN a Student views an answer, THE App SHALL display the summary, then the ECG_Findings grouped by category, then the interpretation_text, then the related_links.

### Requirement 28: Clinical Pearls and References

**User Story:** As a Student, I want to see clinical pearls and academic references for each case, so that I can absorb high-yield teaching points and trace information back to primary sources.

#### Acceptance Criteria

1. THE API SHALL store an array of Clinical_Pearls per Case as individual string entries.
2. THE API SHALL store an array of Case_References per Case, each with a citation string and optional URL.
3. WHEN a Case has Clinical_Pearls, THE App SHALL display the pearls in a distinct "Clinical Pearls" section after the questions.
4. WHEN a Case has Case_References, THE App SHALL display the references with clickable links where URLs are provided.

### Requirement 29: Extended Tagging and Classification System

**User Story:** As a Student, I want cases tagged by ECG topic, difficulty, board relevance, and clinical urgency, so that I can filter and prioritize my study based on exam relevance and clinical importance.

#### Acceptance Criteria

1. THE API SHALL store Case_Tags with a primary_topic field from a defined enum of 25 ECG topics (normal_ecg, sinus_rhythms, atrial_arrhythmias, junctional_rhythms, ventricular_arrhythmias, heart_blocks, bundle_branch_blocks, fascicular_blocks, pre_excitation, acute_coronary_syndromes, stemi_equivalents, st_segment_changes, t_wave_abnormalities, axis_deviation, chamber_enlargement, electrolyte_disturbances, drug_effects, pericardial_disease, cardiomyopathy, pacemaker_ecg, pediatric_ecg, pulmonary_embolism, intervals_and_segments, ecg_artifacts, diagnostic_algorithms).
2. THE API SHALL store optional secondary_topics as an array of strings on each Case_Tag.
3. THE API SHALL store a litfl_category field from a defined enum (ECG, Cardiology, ICE, Toxicology, Metabolic, Resus, Pulmonary, Neurology, Other).
4. THE API SHALL store difficulty (beginner, intermediate, advanced), board_relevance (high, medium, low), and clinical_urgency (emergent, urgent, routine) on each Case_Tag.
5. WHEN a Student filters Cases, THE App SHALL support filtering by primary_topic, difficulty, board_relevance, clinical_urgency, and Source_Type.
6. THE API SHALL store keywords as an array of strings on each Case for search indexing.

### Requirement 30: Extended Media Types and Timing

**User Story:** As a Student, I want ECG media properly categorized by type and timing, so that I can distinguish between 12-lead ECGs, rhythm strips, right-sided leads, and before/after comparisons.

#### Acceptance Criteria

1. THE API SHALL store an extended media type enum on Question_Media: `ecg_12lead`, `ecg_rhythm_strip`, `ecg_right_sided`, `ecg_posterior`, `ecg_single_lead`, `algorithm_diagram`, `clinical_image`, `video`.
2. THE API SHALL store an optional Media_Timing field on Question_Media: `initial`, `post_treatment`, `serial`, `comparison`.
3. THE API SHALL store a media_id string and local_filename string on each Question_Media record for cross-referencing with imported data.
4. WHEN a Sub_Case contains Media with different timing values, THE App SHALL visually distinguish initial ECGs from post-treatment or comparison ECGs.
5. WHEN a Media item has a caption, THE App SHALL display the caption below the media.

### Requirement 31: Case-Based Browsing and Study

**User Story:** As a Student, I want to browse and study cases by source collection, topic, and difficulty, so that I can work through clinical scenarios systematically.

#### Acceptance Criteria

1. THE App SHALL provide a Case browser accessible from the Library page, listing Cases with title, Source_Type, primary_topic, and difficulty.
2. WHEN a Student selects a Case, THE App SHALL display the clinical_context, then present Sub_Cases with their media and progressive Questions.
3. THE App SHALL support filtering the Case browser by Source_Type, primary_topic, difficulty, and board_relevance.
4. THE App SHALL display the total number of Cases and Questions matching the current filter criteria.
5. WHEN a Student completes all Questions in a Case, THE App SHALL display the Clinical_Pearls and Case_References for that Case.

### Requirement 32: Material Design 3 UX Patterns

**User Story:** As a Student, I want the app to follow modern, consistent UX patterns based on Material Design 3, so that the interface feels polished, intuitive, and familiar across all screens.

#### Acceptance Criteria

1. THE App SHALL implement the Material Design 3 (M3) design system as the foundation for all UI components, following the guidelines at https://m3.material.io/.
2. THE App SHALL use the M3 color system with a custom theme generated from a seed color, applying primary, secondary, tertiary, and surface color roles consistently across all components.
3. THE App SHALL use M3 typography scale (display, headline, title, body, label) with appropriate type roles for all text elements throughout the App.
4. THE App SHALL use M3 elevation system with tonal color elevation (surface tint) rather than shadow-only elevation for cards, sheets, and overlays.
5. THE App SHALL use M3 shape system with rounded corners at the appropriate scale tokens (extra-small, small, medium, large, extra-large) for all containers, buttons, and interactive elements.
6. THE App SHALL implement M3 navigation patterns: a Navigation Bar (bottom bar) for mobile with 4 destinations (Dashboard, Study, Library, Progress), and a Navigation Rail for tablet/desktop viewports.
7. THE App SHALL use M3 component patterns for interactive elements including Filled and Outlined buttons, FABs for primary actions, Icon buttons for secondary actions, and Segmented buttons for mode selection (e.g., study session mode picker, answer format selector).
8. THE App SHALL use M3 Cards (filled, elevated, or outlined variants) for content containers including dashboard focus cards, question cards, case browser cards, and analytics summary cards.
9. THE App SHALL use M3 Dialogs for confirmations (e.g., end session, block user) and M3 Bottom Sheets for contextual actions on mobile (e.g., filter options, media expand).
10. THE App SHALL use M3 Chips (filter chips for case/question filtering, assist chips for suggested actions, input chips for keyword tags) throughout the Library and Case Browser.
11. THE App SHALL implement M3 motion and transitions using the standard easing and duration tokens for page transitions, component state changes, and feedback animations (e.g., answer submission, score reveal).
12. THE App SHALL use M3 Text Fields (filled or outlined) with supporting text, error states, and leading/trailing icons for all form inputs including search bars, free-text answer input, and admin forms.
13. THE App SHALL use M3 Progress Indicators (linear for session progress, circular for loading states) and M3 Snackbars for transient feedback messages (e.g., "Bookmark saved", "Answer submitted").
14. THE App SHALL implement M3 state layers (hover, focus, pressed, dragged) on all interactive elements to provide consistent visual feedback across input methods.
